# 클라우드 서비스 도입 시 보안 고려 요소

## 1. 개요

### 가. 정의
> 기업이 클라우드 서비스를 도입할 때 **데이터·접근·규제·운영 전반에서 검토·확보해야 할 보안 요소**. 온프레미스와 다른 클라우드 특유의 위험에 대응하기 위한 것이다.

클라우드 보안이 어려운 근본 이유는 '**통제권의 일부가 클라우드 사업자(CSP)로 넘어가는**' 책임공유모델에 있다. 온프레미스에서는 기업이 물리 서버부터 데이터까지 모든 계층을 직접 통제했지만, 클라우드에서는 물리 인프라·하이퍼바이저는 CSP가, 데이터·계정·접근·설정은 이용자가 책임진다. 문제는 이 경계를 오해할 때 발생한다. "클라우드니까 CSP가 알아서 보안을 챙기겠지"라고 방심하면, 정작 이용자 책임인 접근 권한 관리나 저장소 설정을 소홀히 해 사고가 난다. 실제로 대부분의 클라우드 유출 사고는 CSP의 인프라 해킹이 아니라, 이용자의 **설정 오류(공개된 스토리지 버킷)나 계정 탈취** 에서 비롯된다. 따라서 클라우드 보안의 출발점은 "내가 무엇을 책임지는가"를 정확히 아는 것이다.

### 나. 필요성
클라우드는 확장성·비용 이점을 주지만, 인터넷에 노출되고 다수가 공유하는 특성상 새로운 공격 표면을 만든다. 도입 전 보안 요소를 체계적으로 검토해야 이점을 안전하게 누릴 수 있다.

## 2. 책임공유모델

```mermaid

  C["CSP 책임<br/>(물리·인프라·하이퍼바이저)"] --- U["이용자 책임<br/>(데이터·계정·접근·설정)"]
  style U fill:#e8f0fe,stroke:#2f6fed,stroke-width:2px
```

책임공유모델은 서비스 모델(IaaS·PaaS·SaaS)에 따라 경계가 이동한다. IaaS는 이용자가 OS·미들웨어까지 책임지고, SaaS는 데이터·계정만 책임지는 식이다. 어느 경우든 **데이터와 계정 보안은 항상 이용자 몫** 이라는 점이 핵심이다.

## 3. 주요 보안 고려 요소

클라우드 도입 시에는 데이터부터 운영까지 여러 요소를 검토해야 한다. 데이터는 저장·전송 시 암호화하고 위치·주권을 확인하며, 접근은 최소권한·MFA로 통제한다. 잘못된 구성을 막고(CSPM), 망분리·보안그룹으로 네트워크를 보호하며, 로그·감사로 가시성을 확보한다. 규제(CSAP·개인정보)를 준수하고, 가용성·재해복구와 종료·전환(Exit) 전략, 컨테이너·이미지 등 공급망 보안까지 살핀다.

| 요소 | 고려 사항 |
|---|---|
| **데이터 보안** | 암호화(저장·전송), 데이터 위치·주권, 백업·파기 |
| **접근 통제(IAM)** | 최소권한, MFA, 계정·키 관리 |
| **설정 보안** | 잘못된 구성 방지(CSPM) |
| **네트워크** | 망분리·보안그룹·방화벽, 제로트러스트 |
| **가시성·모니터링** | 로그·감사, 위협 탐지 |
| **규제·인증** | CSAP·ISMS, 개인정보·산업 규제 준수 |
| **연속성** | 가용성·DR, Exit(종료·전환) 전략 |
| **공급망** | 컨테이너·이미지·IaC 보안, SBOM |

## 4. 고려사항 및 시사점

1. **책임공유모델의 이해가 보안의 출발점**이다. 선택한 서비스 모델에서 자신이 책임지는 영역(항상 데이터·계정 포함)을 명확히 인지하고 그에 맞는 보안을 갖춰야 한다.
2. **설정 오류·계정 관리가 최대 위험**이다. 공개된 스토리지, 과도한 권한, 유출된 액세스 키가 실제 사고의 다수를 차지하므로, CSPM(설정 점검)과 IAM(권한 관리)을 강화한다.
3. **CNAPP·제로트러스트로 통합 관리**하는 방향으로 발전한다. 클라우드 네이티브 보안 플랫폼으로 워크로드·설정·권한을 통합 점검하고, 내부도 신뢰하지 않는 제로트러스트로 접근을 통제한다.

---

> **한 줄 요약**: 클라우드 도입 시 *책임공유모델을 기반으로 데이터 암호화·IAM·설정보안·모니터링·규제준수·연속성* 을 검토해야 하며, 설정 오류·계정 관리가 최대 위험이므로 CSPM·IAM·제로트러스트로 통합 관리한다.

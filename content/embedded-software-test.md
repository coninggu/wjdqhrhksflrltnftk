# 임베디드 소프트웨어 테스트(Embedded Software Test)

## 1. 개요

### 가. 정의
> **임베디드 소프트웨어 테스트**는 특정 하드웨어에 내장되어 실시간·자원제약 환경에서 동작하는 임베디드 SW의 **기능·성능·안전성을 하드웨어와의 상호작용까지 포함해 검증**하는 활동이다.

임베디드 SW 테스트가 일반 SW 테스트와 다른 근본 이유는 '**소프트웨어가 하드웨어와 물리 세계에 묶여 있다**'는 데 있다. 일반 애플리케이션은 PC·서버라는 풍부한 환경에서 돌지만, 임베디드 SW는 제한된 메모리·CPU를 가진 특정 하드웨어(자동차 ECU·의료기기·가전)에 내장되어, 센서·액추에이터와 실시간으로 상호작용한다. 그래서 테스트도 소프트웨어만 볼 수 없다. 하드웨어 타이밍, 실시간 응답(마감시간 준수), 인터럽트, 자원 제약, 그리고 오동작이 곧 물리적 사고(차량 급발진·의료 오작동)로 이어지는 안전성까지 검증해야 한다. 게다가 대상 하드웨어에서 직접 테스트하기 어렵고 디버깅 수단이 제한적이라, 시뮬레이션·에뮬레이션·실장비를 단계적으로 활용하는 독특한 테스트 전략(MIL→SIL→PIL→HIL)이 필요하다.

### 나. 특수성
실시간성, 자원 제약, 하드웨어 의존성, 높은 안전·신뢰 요구, 제한된 관찰·디버깅 수단이 임베디드 테스트의 난도를 높인다.

## 2. 테스트 단계(X-in-the-Loop)

```mermaid
flowchart LR
  MIL["MIL<br/>(모델)"] --> SIL["SIL<br/>(SW)"] --> PIL["PIL<br/>(프로세서)"] --> HIL["HIL<br/>(HW 통합)"]
  style HIL fill:#e8f0fe,stroke:#2f6fed,stroke-width:2px
```

임베디드 테스트는 개발 단계에 맞춰 점진적으로 실제 환경에 가까워진다. 초기엔 모델·시뮬레이션으로 검증하고, 점차 실제 프로세서·하드웨어를 결합해 최종적으로 실장비에서 통합 검증한다.

| 단계 | 내용 |
|---|---|
| **MIL(Model-in-the-Loop)** | 모델 수준 시뮬레이션 검증 |
| **SIL(Software-in-the-Loop)** | 실제 코드를 PC 환경에서 검증 |
| **PIL(Processor-in-the-Loop)** | 목표 프로세서에서 코드 검증 |
| **HIL(Hardware-in-the-Loop)** | 실제 하드웨어와 통합 실시간 검증 |

## 3. 주요 테스트 관점

| 관점 | 내용 |
|---|---|
| **실시간성** | 마감시간(deadline) 준수, 응답 시간 |
| **자원 제약** | 메모리·CPU·전력 한계 내 동작 |
| **하드웨어 연동** | 센서·인터럽트·통신 정확성 |
| **안전성·신뢰성** | 고장 시 안전(fail-safe), 커버리지 |
| **환경 시험** | 온도·전압·노이즈 등 물리 조건 |

## 4. 고려사항 및 시사점

1. **안전 표준 준수가 필수**다. 자동차(ISO 26262)·의료기기·항공(DO-178C) 등 도메인별 기능안전 표준이 요구하는 테스트 커버리지(MC/DC 등)와 절차를 충족해야 한다. [[software-safety-analysis]]
2. **시뮬레이션과 실장비의 균형**이 중요하다. 초기엔 SIL·MIL로 빠르고 안전하게 대량 검증하고, 최종 단계엔 HIL·실차 시험으로 실제 환경을 확인하는 단계적 접근이 비용·품질을 최적화한다.
3. **자동화·CI 적용이 확산**된다. 임베디드에서도 테스트 자동화·지속통합(CI)과 HIL 자동화 장비를 결합해, 잦은 변경에도 회귀 검증을 신속히 수행하는 방향으로 발전하고 있다.

---

> **한 줄 요약**: 임베디드 SW 테스트는 *하드웨어·실시간·자원제약·안전성까지 포함해 검증* 하는 활동으로, MIL→SIL→PIL→HIL로 점진적으로 실제 환경에 접근하며 ISO 26262 등 기능안전 표준 준수가 핵심이다.

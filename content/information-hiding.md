# 정보은닉(Information Hiding)

## 1. 개요

### 가. 정의
> 모듈 내부의 **구현 세부사항·데이터를 외부로부터 감추고**, 잘 정의된 인터페이스만 공개하여 모듈 간 결합도를 낮추는 소프트웨어 설계 원리. D.L. Parnas(1972)가 제안.

### 나. 필요성
- 내부 변경이 외부에 미치는 **파급 효과(Ripple Effect) 최소화**
- 모듈 **독립성·유지보수성** 향상
- 복잡도 은폐를 통한 **추상화** 실현

- 외부는 **무엇(What)** 을 하는지만 알고, **어떻게(How)** 하는지는 알 필요 없음

## 3. 정보은닉과 캡슐화

| 구분 | 정보은닉 (Information Hiding) | 캡슐화 (Encapsulation) |
|---|---|---|
| **개념** | 구현 세부를 감추는 **설계 원리** | 데이터+메서드를 하나로 묶는 **기법** |
| **초점** | 접근 제한·은폐 | 결합(묶음) |
| **관계** | 캡슐화를 통해 정보은닉을 **실현** | 정보은닉을 지원하는 수단 |
| **구현 예** | `private` 접근제어자, 모듈 인터페이스 | 클래스, 객체 |

## 4. 장점 및 효과

| 효과 | 설명 |
|---|---|
| **낮은 결합도** | 모듈 간 의존성 감소 → 독립적 수정 가능 |
| **높은 응집도** | 관련 기능이 모듈 내부에 집중 |
| **유지보수성** | 내부 변경이 인터페이스 유지 시 외부에 무영향 |
| **재사용성·보안** | 검증된 인터페이스 재사용, 내부 데이터 보호 |

## 5. 시사점
- 객체지향의 **캡슐화·추상화**, MSA의 **서비스 경계**, API 설계의 기본 원리
- 인터페이스 안정성이 곧 시스템 유연성 → **인터페이스 우선 설계(API-first)**

---

> **한 줄 요약**: 정보은닉은 모듈의 *내부 구현을 감추고 인터페이스만 공개* 하여 결합도를 낮추는 설계 원리로, **캡슐화를 통해 실현**되어 유지보수성과 독립성을 높인다.

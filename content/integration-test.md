# 통합 테스트(Integration Test)

## 1. 개요

### 가. 정의
> 단위 테스트를 마친 **모듈들을 결합했을 때 인터페이스와 상호작용이 올바른지** 검증하는 테스트. 모듈 간 데이터 전달·연동에서 발생하는 오류를 찾는 것이 목적이다.

통합 테스트가 단위 테스트와 별개로 반드시 필요한 이유는 '**각 부품이 정상이어도 조립하면 문제가 생긴다**'는 데 있다. 개별 모듈이 각자의 테스트를 완벽히 통과했더라도, 막상 합치면 인터페이스 규약의 미세한 불일치, 데이터 형식·단위의 차이, 호출 순서나 타이밍 문제로 오류가 발생한다. 단위 테스트가 '부품 검사'라면 통합 테스트는 '조립 검사'다. 예를 들어 A모듈이 날짜를 'YYYYMMDD'로 넘기는데 B모듈이 'YYYY-MM-DD'를 기대한다면, 각자는 정상이지만 결합하면 실패한다. 이런 결함은 결합해봐야만 드러난다.

### 나. 필요성
결함은 발견이 늦을수록 수정 비용이 커진다. 모듈 간 인터페이스 오류를 통합 단계에서 조기에 잡지 못하면 시스템 테스트나 운영에서 발견되어 비용이 급증하므로, 체계적인 통합 테스트가 필수다.

## 2. 통합 방식: 비점진적 vs 점진적

```mermaid
flowchart TB
  I[통합 테스트] --> N["비점진적<br/>(빅뱅)"]
  I --> P[점진적]
  P --> P1[하향식 Top-Down]
  P --> P2[상향식 Bottom-Up]
  P --> P3[샌드위치 혼합]
  style I fill:#e8f0fe,stroke:#2f6fed,stroke-width:2px
```

통합 방식은 모듈을 한꺼번에 합치느냐, 단계적으로 합치느냐로 나뉜다. **비점진적(빅뱅)** 은 모든 모듈을 한 번에 결합해 테스트하는 방식으로, 준비는 간단하지만 오류가 나면 어느 모듈 간 문제인지 원인 격리가 매우 어렵다. **점진적** 은 모듈을 하나씩 추가하며 테스트하는 방식으로, 시간은 더 걸리지만 새로 추가한 모듈에서 문제가 드러나므로 오류 격리가 쉽다. 그래서 실무에서는 점진적 방식이 선호된다.

| 방식 | 설명 | 특징 |
|---|---|---|
| **비점진적(빅뱅)** | 모든 모듈 한 번에 결합 | 준비 간단, 원인 파악 곤란 |
| **점진적** | 모듈을 단계적 추가 | 오류 격리 용이, 시간 소요 |

## 3. 하향식 vs 상향식

점진적 통합은 결합 방향에 따라 하향식과 상향식으로 나뉜다. **하향식(Top-Down)** 은 최상위 모듈부터 시작해 하위로 내려가며 결합하는데, 아직 완성되지 않은 하위 모듈 자리에 가짜 응답을 주는 **스텁(Stub)** 이 필요하다. 상위의 제어·설계 결함을 조기에 발견하는 장점이 있다. **상향식(Bottom-Up)** 은 최하위 모듈부터 시작해 위로 올라가며 결합하는데, 아직 없는 상위 모듈 대신 하위를 호출해주는 **드라이버(Driver)** 가 필요하다. 하위 모듈을 철저히 검증하는 장점이 있다.

| 구분 | 하향식(Top-Down) | 상향식(Bottom-Up) |
|---|---|---|
| **순서** | 상위 → 하위 | 하위 → 상위 |
| **필요 도구** | 스텁(Stub) | 드라이버(Driver) |
| **장점** | 상위 설계 결함 조기 발견 | 하위 모듈 철저 검증 |
| **단점** | 하위 미완성 시 스텁 다수 | 상위 결함 늦게 발견 |

## 4. 테스트 드라이버와 테스트 스텁

두 보조 모듈의 차이는 방향이다. **드라이버** 는 아직 만들어지지 않은 상위 모듈을 대신해 하위 모듈을 호출하고 테스트 데이터를 넣어 결과를 확인하는 '가짜 상위'다. **스텁** 은 아직 만들어지지 않은 하위 모듈을 대신해, 상위 모듈의 호출에 최소한의 더미 응답을 돌려주는 '가짜 하위'다. 상향식은 하위부터 검증하므로 드라이버를, 하향식은 상위부터 검증하므로 스텁을 사용한다.

| 구분 | 테스트 드라이버 | 테스트 스텁 |
|---|---|---|
| **역할** | 하위를 호출하는 '가짜 상위' | 호출에 응답하는 '가짜 하위' |
| **사용** | 상향식 | 하향식 |
| **기능** | 테스트 데이터 입력·결과 확인 | 더미 응답 반환 |

## 5. 고려사항 및 시사점

1. **실무는 샌드위치(혼합) 통합**이 일반적이다. 하향식과 상향식을 동시에 진행해 중간에서 만나는 방식으로, 양쪽 장점을 취하되 드라이버·스텁이 모두 필요하다.
2. **CI 파이프라인에 통합 테스트 자동화**를 내재화한다. 코드 변경마다 자동으로 통합 테스트를 돌려 회귀를 조기에 잡는다.
3. **MSA에서는 계약 테스트(Contract Test)로 확장**된다. 서비스 간 API 규약을 계약으로 정의하고 각 서비스가 이를 준수하는지 독립 검증해, 분산 환경의 통합 오류를 관리한다.

---

> **한 줄 요약**: 통합 테스트는 모듈 결합 시 인터페이스 오류를 검증하며, *비점진적(빅뱅)·점진적(하향식·상향식)* 방식으로 수행하고 하향식은 스텁을, 상향식은 드라이버를 사용하며 실무에서는 샌드위치 혼합과 CI 자동화·계약 테스트로 발전한다.

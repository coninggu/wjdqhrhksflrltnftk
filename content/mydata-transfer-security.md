# 마이데이터 전송 보안 (마이데이터 전송 보안 안내서, 2023.09)

## 1. 개요

### 가. 정의·배경
> 마이데이터(본인신용정보관리업)에서 **정보주체의 요구에 따라 개인(신용)정보를 기관 간 전송할 때 발생하는 위험을 통제하기 위한 안전조치 기준**을 정리한 안내서로, 전송자·수신자 양측의 보호책임자 지정·접근관리·관리적 안전조치를 요구한다.

마이데이터의 본질은 여러 기관에 흩어진 내 정보를 **정보주체의 전송요구권**에 근거해 한곳으로 모아 활용하는 것이다. 이 과정에서 은행·카드·통신사 등 정보제공자와 마이데이터 사업자 사이에 대량의 민감한 개인정보가 **API를 통해 상시 오간다**는 점이 핵심 위험이다. 데이터가 이동하는 순간이 곧 유출·위·변조·오전송의 위험 구간이므로, 전송 전 구간을 안전하게 통제하는 것이 마이데이터 신뢰의 전제가 된다.

### 나. 필요성
마이데이터가 확산되면서 전송되는 정보의 양과 참여 기관 수가 폭증했고, 한 기관의 보안 허점이 전체 생태계의 신뢰를 무너뜨릴 수 있는 구조가 되었다. 특히 자산·소득·신용 같은 **재산 관련 민감정보**가 대상이라 유출 시 피해가 직접적이고 회복이 어렵다. 이에 개인정보보호위원회는 신용정보법·개인정보보호법의 안전조치 의무를 마이데이터 전송 맥락에 맞게 구체화한 안내서를 통해, 전송자와 수신자가 **동일한 수준의 보호책임**을 지도록 기준을 통일했다.

## 2. 전송대상 개인정보 보호책임자(CPO) 지정

안전조치는 결국 "누가 책임지고 관리하는가"에서 출발한다. 기술적 통제를 아무리 갖춰도 이를 총괄·점검할 주체가 없으면 사고 시 대응이 표류하기 때문이다. 안내서는 전송 업무를 관장하는 **개인정보 보호책임자(CPO)** 를 명확히 지정해 전송 보안 정책의 수립과 이행 점검, 침해사고 대응을 총괄하도록 한다. 이때 CPO는 실효적 권한과 **독립성**을 갖고 경영진에 직접 보고할 수 있어야 하는데, 현업 부서에 종속되면 편의를 위해 보안 통제를 완화하려는 압력에 저항하기 어렵기 때문이다.

| 항목 | 내용 |
|---|---|
| **CPO 지정** | 전송 업무 총괄 보호책임자 지정·책임 명확화 |
| **역할** | 전송 보안 정책 수립·이행 점검, 침해사고 대응 총괄 |
| **독립성** | 실질 권한·독립성 부여, 경영진 직보 체계 |

## 3. 전송대상 개인정보처리시스템 접근 관리

```mermaid
flowchart LR
  A[접근권한 최소화] --> B[인증 강화]
  B --> C[접속기록 관리]
  C --> D[이상행위 탐지]
```

접근 관리의 원리는 **"필요한 사람이, 필요한 만큼만, 흔적을 남기고"** 로 요약된다. 개인정보처리시스템에 대한 권한을 최소한으로 부여하고 직무를 분리(개발자와 운영자 분리 등)하는 이유는, 한 계정이 탈취되더라도 접근 가능한 정보의 범위를 좁혀 피해를 국한하기 위해서다. 여기에 비밀번호 단일 인증은 도용에 취약하므로 **다요소 인증(MFA)** 으로 계정 도용 위험을 낮추고, 모든 접속·처리 행위를 **접속기록으로 남겨 위·변조 방지 조치**와 함께 보관한다. 접속기록은 사후 추적의 근거일 뿐 아니라, 심야 대량 조회 같은 **이상행위를 실시간 탐지**하는 입력이 되어 사고를 조기에 차단한다. 예컨대 한 계정이 평소의 수백 배에 달하는 전송 요청을 보내면 자동 차단·알림이 발동하도록 설계한다.

| 항목 | 내용 |
|---|---|
| **접근권한** | 최소권한·직무분리, 권한 부여·회수의 주기적 검토 |
| **인증** | 다요소 인증 등 안전한 인증수단, 계정·세션 관리 |
| **접속기록** | 접속·처리기록 보관 및 위·변조 방지 |
| **통제·탐지** | 접근통제 시스템, 이상행위 모니터링·자동 차단 |

## 4. 개인정보 관리 및 재해·재난 대비

전송 데이터의 안전은 **이동 중(in transit)과 저장 중(at rest)** 양쪽을 모두 지켜야 완성된다. 전송 구간은 TLS로, 저장 데이터는 암호화하여 중간자 공격이나 저장매체 탈취 시에도 내용을 보호한다. 나아가 데이터가 전달 과정에서 바뀌지 않았음을 보장하는 **무결성 검증**과 유출 시도를 탐지·차단하는 DLP 성격의 통제를 둔다. 한편 마이데이터는 다수 기관이 실시간으로 연결된 서비스라 **가용성 자체가 보안 요구**가 된다. 특정 기관의 시스템이 재해로 멈추면 전송 체인 전체가 영향을 받으므로, 백업·복구 체계(BCP/DRS)와 이중화로 서비스 연속성을 확보하고, 침해사고 발생 시 신속한 대응·신고 절차를 사전에 마련해 둔다.

| 항목 | 내용 |
|---|---|
| **암호화** | 전송구간(TLS)·저장 데이터 암호화 |
| **무결성·유출방지** | 전송 데이터 위·변조 방지, 유출 탐지·차단(DLP) |
| **재해·재난 대비** | 백업·복구(BCP/DRS), 시스템 이중화로 연속성 확보 |
| **사고 대응** | 침해사고 대응 절차·신고 체계 수립 |

## 5. 고려사항 및 시사점(기술사 관점)
- **전송 전 구간의 신뢰 사슬**: 인증·암호화·접근통제 어느 한 곳이 약하면 전체가 무너지므로, 상호인증(mTLS) 기반 API 보안과 결합해 종단 간 신뢰를 확보한다.
- **표준 API와 보안의 정합**: 마이데이터 표준 API(기능적합성·보안적합성 심사)와 본 안내서의 안전조치를 함께 충족해야 하며, 접근토큰의 최소 권한·짧은 유효기간 설계가 실무 관건이다.
- **법·제도 준수**: 신용정보법·개인정보보호법의 안전성 확보조치 기준과 정합하며, 정기 점검·임직원 교육으로 관리적 통제의 실효성을 유지한다.
- **생태계 신뢰가 곧 경쟁력**: 참여 기관 전체가 동일 수준의 보안을 지켜야 하므로, 취약한 참여자를 걸러내는 인증·감독 체계가 마이데이터 확산의 전제가 된다.

---

> **한 줄 요약**: 마이데이터 전송 보안 안내서는 *보호책임자(CPO) 지정, 처리시스템 접근관리(최소권한·다요소인증·접속기록), 개인정보 관리·재해대비(암호화·무결성·백업·사고대응)* 를 통해 기관 간 전송되는 민감정보의 신뢰 사슬을 확보하도록 요구한다.

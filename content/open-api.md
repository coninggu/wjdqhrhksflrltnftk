# 개방형 API(Open API)

## 1. 개요

### 가. 정의
> 외부 개발자·서비스가 접근·이용할 수 있도록 **공개된 표준 인터페이스**로, 조직이 보유한 데이터·기능을 개방해 서비스 연계·확장과 **생태계(플랫폼)** 를 촉진한다.

### 나. 등장 배경 및 필요성
과거 시스템은 폐쇄적이라 외부와 데이터를 주고받으려면 매번 개별 연동을 개발해야 했다. 하지만 디지털 경제에서는 **하나의 서비스가 여러 서비스와 결합(매시업)** 할 때 가치가 커진다. 지도 위에 배달·부동산 정보를 얹는 것처럼 말이다. Open API는 이 결합을 "표준화된 계약"으로 만들어, 누구나 정해진 규격에 따라 기능을 호출할 수 있게 한다. 특히 금융권의 **오픈뱅킹·마이데이터**는 법·제도로 API 개방을 강제해, Open API가 산업 인프라가 된 대표 사례다.

### 다. 특징

Open API의 특징은 곧 그 가치이자 관리 부담이다. 개방·표준·재사용성이 생태계를 키우는 반면, 아무나 호출할 수 있으므로 **인증·과금·트래픽 통제**가 반드시 수반된다.

| 특징 | 내용 |
|---|---|
| 개방성 | 인증된 외부 주체에 기능·데이터 공개 |
| 표준성 | HTTP·REST·OpenAPI(Swagger) 등 표준 준수 |
| 재사용성 | 매시업·플랫폼 생태계 확장 |
| 관리 필요 | 인증·과금·버전·트래픽 통제(API 게이트웨이) |

## 2. SOAP와 REST 구성요소 비교

Open API를 구현하는 두 방식이 SOAP와 REST다. 둘의 근본 차이는 SOAP은 **엄격한 XML 프로토콜**이고 REST는 **HTTP를 그대로 활용하는 아키텍처 스타일**이라는 점이다. SOAP은 표준이 강하고 WS-Security·트랜잭션을 갖춰 은행 간 거래처럼 높은 신뢰성이 필요한 곳에 맞지만, 무겁다. REST는 자원을 URI로 표현하고 HTTP 메서드(GET·POST 등)로 다뤄 가볍고 캐싱·확장이 쉬워, 웹·모바일·공개 API의 사실상 표준이 됐다. 이 때문에 오늘날 Open API 대부분은 REST(또는 그 대안인 GraphQL)로 제공된다.

| 구분 | SOAP | REST |
|---|---|---|
| 개념 | XML 기반 프로토콜 | HTTP 기반 아키텍처 스타일 |
| 구성요소 | Envelope·Header·Body, WSDL, UDDI | 자원(URI)·HTTP 메서드·표현(JSON)·무상태 |
| 메시지 | XML | 주로 JSON |
| 특징 | 강한 표준·트랜잭션·WS-Security | 경량·확장성·캐싱·Stateless |
| 적합 | 엔터프라이즈·고신뢰 | 웹·모바일·공개 API |

```mermaid
  C[클라이언트] -->|HTTP 요청| G[API 게이트웨이]
  G -->|인증·라우팅·Rate Limit| S[API 서버 · 자원]

```

## 3. 취약점 및 대응 방안 (OWASP API Top 10)

Open API는 외부에 노출되므로 웹 애플리케이션과는 다른 API 특유의 위협을 받는다. 그중 가장 흔하고 치명적인 것이 **BOLA(객체 수준 인가 미흡)** 인데, 인증은 통과했지만 "**남의 데이터에 접근할 권한**"까지 검증하지 않아 URL의 ID만 바꾸면 타인 정보가 조회되는 결함이다. 예컨대 `/orders/1001`을 `/orders/1002`로 바꿔 남의 주문을 보는 식이다. 따라서 요청마다 **해당 객체의 소유·권한을 서버가 검증**해야 한다.

| 취약점 | 원리 | 대응 |
|---|---|---|
| 인증·인가 미흡(BOLA) | 객체 소유권 미검증 | OAuth 2.0·JWT + 객체 수준 권한 검증 |
| 과도한 데이터 노출 | 응답에 불필요 필드 포함 | 응답 필드 최소화, 스키마 검증 |
| 자원 고갈(DoS) | 무제한 호출·대량 조회 | Rate Limiting·쿼터, 페이지네이션 |
| 인젝션 | 미검증 입력이 쿼리로 실행 | 입력 검증, 파라미터 바인딩 |
| 부적절한 자산관리 | 방치·구버전 API 노출 | API 인벤토리·버전 관리, 폐기 API 차단 |

## 4. API 관리(수명주기)

Open API는 한 번 공개하면 끝이 아니라, 외부 이용자가 의존하므로 **버전·폐기까지 신중히 관리**해야 한다. 갑자기 API를 바꾸면 이를 쓰던 모든 서비스가 깨지기 때문이다. 그래서 설계 단계에서 **OpenAPI 명세로 계약을 먼저 정하고(계약 우선)**, 게이트웨이·포털로 게시·운영하며, 폐기 시에는 충분한 유예와 마이그레이션 안내를 둔다.

| 단계 | 활동 |
|---|---|
| 설계 | OpenAPI 명세(계약 우선), 표준화 |
| 게시 | API 게이트웨이·개발자 포털 등록 |
| 운영 | 인증·과금·모니터링·트래픽 제어 |
| 폐기 | 버전 폐기·마이그레이션 안내 |

## 5. 고려사항 및 시사점
- **게이트웨이 중심 통제**: 인증·트래픽 제한·버전·로깅을 개별 API마다 구현하는 대신 **API 게이트웨이**로 일원화해, 보안·운영을 일관되게 적용한다.
- **계약 우선(Contract First)**: OpenAPI 명세를 먼저 확정하면 문서·목서버·클라이언트 코드를 자동 생성할 수 있어 개발 생산성과 일관성이 오른다.
- **보안 강화·전망**: 내부망도 신뢰하지 않는 **제로트러스트**와 **mTLS**로 API 구간을 보호하며, Open API는 오픈뱅킹·마이데이터·공공데이터 개방의 기반 인프라로 계속 확대되고 있다.

---

> **한 줄 요약**: 개방형 API는 표준(REST/SOAP) 기반 공개 인터페이스로 매시업·플랫폼 생태계를 확장하며, OAuth·객체 권한 검증·게이트웨이·Rate Limiting으로 OWASP API 취약점(특히 BOLA)에 대응하고, 계약 우선·수명주기 관리로 운영하는 오픈뱅킹·마이데이터의 기반이다.

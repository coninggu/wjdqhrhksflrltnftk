# 오픈소스 라이선스 정책 변화 (개방형 → 폐쇄형)

## 1. 개요

### 가. 현상
> MongoDB·Elastic·HashiCorp·Redis 등 주요 오픈소스가 **개방형(MIT·BSD·Apache)에서 소스 공개형 제한 라이선스(SSPL·BSL)로 전환**하는 흐름. 클라우드 시대의 수익화·지속가능성 문제에서 비롯된다.

### 나. 라이선스 유형 비교

| 구분 | 개방형 | 폐쇄형(소스공개 제한) |
|---|---|---|
| **예시** | MIT, BSD, Apache 2.0 | SSPL, BSL, Elastic License |
| **특징** | 상업적 이용·수정·재배포 자유 | 서비스형(SaaS) 제공·경쟁 이용 제한 |
| **OSI 승인** | 승인(오픈소스) | 미승인(소스 공개형) |

## 2. 정책 변경 배경

```mermaid
flowchart LR
  A[클라우드 사업자<br/>무임승차·SaaS화] --> B[원 개발사 수익 악화]
  B --> C[지속가능성·투자회수 위기]
  C --> D[라이선스 전환<br/>SSPL·BSL]
```

| 배경 | 내용 |
|---|---|
| **클라우드 무임승차** | 하이퍼스케일러가 OSS를 관리형 서비스로 제공해 수익 독식 |
| **수익화·지속가능성** | 개발 지속 위한 재원 확보 |
| **경쟁 방어** | 동일 서비스로의 상업적 재판매 제한 |

## 3. 소프트웨어 산업 영향

| 영향 | 내용 |
|---|---|
| **커뮤니티 반발·포크** | OpenSearch(ES), Valkey(Redis), OpenTofu(Terraform) 등 포크 |
| **기업 이용 리스크** | 라이선스 조항 검토 부담, 벤더 종속·비용 증가 |
| **오픈소스 신뢰성** | '오픈소스' 정의(OSI) 논쟁 |
| **재단화·대체재** | 중립 재단 이관, 대체 프로젝트 활성화 |

## 4. 기업 대응 방안

| 대응 | 내용 |
|---|---|
| **라이선스 인벤토리** | 사용 OSS·라이선스 조항 파악(SCA·SBOM) |
| **리스크 평가** | 변경 가능성·SaaS 제공 여부 검토 |
| **대안 확보** | 포크·대체재·상용 계약 검토 |

## 5. 고려사항 및 시사점
- **오픈소스 지속가능성**(개발 재원)과 **개방성**의 균형 과제
- 도입 기업은 라이선스 변경 리스크를 아키텍처·조달에 반영
- OSI 승인 여부·포크 생태계 동향 지속 모니터링

---

> **한 줄 요약**: 클라우드 무임승차로 인한 수익성 악화가 *개방형→SSPL·BSL 폐쇄형* 전환을 촉발했으며, 포크 확산·이용 리스크에 대해 기업은 SBOM·대체재로 대응해야 한다.

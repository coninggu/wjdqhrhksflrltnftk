# 공공데이터베이스 표준화 관리 (예방적 품질관리, 2023.04)

## 1. 개요

### 가. 정의 및 배경
> 행정안전부가 **고품질 공공데이터의 제공·개방·활용**을 위해 「공공데이터베이스 표준화 관리 매뉴얼」(2023.04)로 제시한, 시스템 **구축 단계부터 오류를 예방**하는 사전(예방적) 품질관리 기준.

공공데이터는 국민 서비스·정책 결정·민간 활용(마이데이터·AI 학습)의 원천이므로, 데이터 하나의 오류가 여러 기관·서비스로 **확산·증폭**된다. 종래의 품질관리가 데이터가 이미 쌓인 뒤 오류를 찾아 고치는 **사후(교정적) 방식**에 치우쳐 있었다면, 이 매뉴얼은 표준을 설계·구축 시점에 내재화해 **오류가 발생하기 전에 차단**하는 방향으로 무게중심을 옮긴 것이 핵심이다.

### 나. 필요성
사후 정제는 이미 잘못 입력·연계된 데이터를 추적해 되돌려야 하므로 비용이 크고, 이미 배포·연계된 데이터의 오류는 회수도 어렵다. 예컨대 표준용어 없이 부서마다 "주민등록번호/주민번호/주민등록No"를 제각각 쓰면, 이후 연계·통합 때마다 매핑·정제 작업이 반복된다. 표준단어·도메인·용어를 **분석 단계에서 미리 확정**하면 이런 반복 비용과 정합성 오류를 원천 차단할 수 있어, "구축 단계부터 품질을 내재화"하는 예방적 접근이 요구된다.

## 2. 시스템 구축 단계별 예방적 품질관리 활동

```mermaid
flowchart LR
  A[분석] --> D[설계] --> I[구현] --> T[시험·이행] --> O[운영]
```

품질관리 활동을 SDLC 전 단계에 배치하는 이유는, **오류를 늦게 발견할수록 수정 비용이 기하급수적으로 커지기 때문**이다. 요구·표준을 분석 단계에서 못 잡으면 설계·구현을 거치며 오류가 코드와 데이터에 굳어져, 운영 단계에서는 정제·재구축이라는 값비싼 대가를 치른다. 그래서 각 단계마다 다음과 같은 방어선을 둔다.

| 단계 | 예방적 품질관리 활동 | 목적 |
|---|---|---|
| **분석** | 데이터 요구·표준 정의, 표준단어·표준도메인 사전 수립 | 표준 부재로 인한 명명·의미 혼란 예방 |
| **설계** | 표준 준수 데이터 모델링, 무결성·제약 규칙 설계 | 모델 정합성·참조무결성 사전 확보 |
| **구현** | 표준 반영 DB 구축, 제약조건(NOT NULL·FK·CHECK) 적용 | 잘못된 값의 입력 자체를 차단 |
| **시험·이행** | 품질 진단·검증, 데이터 정제·이관 정합성 검증 | 이관 과정의 유실·왜곡 방지 |
| **운영** | 지속 품질 모니터링·개선, 정기 진단 | 운영 중 품질 저하(드리프트) 관리 |

특히 **구현 단계의 제약조건**은 예방적 통제의 대표 사례다. 컬럼에 도메인·CHECK 제약을 걸어두면 유효범위를 벗어난 값이 애초에 저장되지 못하므로, 사후에 이상값을 찾아 지우는 작업 자체가 불필요해진다.

## 3. 예방적 품질관리 4개 진단영역·9개 진단항목

진단은 "표준이 제대로 정의되었는가 → 그 표준이 모델에 반영되었는가 → 실제 값과 구조가 표준·규칙을 지키는가"라는 **상류에서 하류로의 흐름**을 따라 4개 영역으로 구성된다. 상류(표준화)가 무너지면 하류(값·구조) 진단이 아무리 촘촘해도 근본 오류를 막지 못하므로, 영역 간에는 의존 관계가 있다.

| 진단영역 | 대표 진단항목 | 무엇을 보장하나 |
|---|---|---|
| **표준화** | 표준단어, 표준도메인, 표준용어 | 용어·의미의 일관성 |
| **모델 품질** | 데이터 모델 정합성, 명명 규칙 준수 | 표준의 모델 반영·구조 일관성 |
| **값 품질** | 필수값(NOT NULL)·유효값(도메인) 준수 | 실제 저장 값의 정확성·완전성 |
| **구조·무결성** | 참조 무결성, 코드 정합성 | 관계·코드 값의 정합성 |

> 위 4개 영역 아래 총 **9개 진단항목**(표준단어·표준도메인·표준용어, 데이터모델 정합성·명명규칙, 필수값·유효값, 참조무결성·코드 정합성 등)으로 사전 진단을 수행한다.

예를 들어 코드 정합성 진단은 "성별 코드 컬럼에 표준 코드셋(1/2/9) 외의 값이 없는가"를 보는데, 이를 표준용어·표준도메인 정의와 연결해 점검함으로써 값·구조·표준이 서로 어긋나지 않도록 교차 검증한다.

## 4. 기대 효과

| 효과 | 내용 | 근거 |
|---|---|---|
| **품질 향상** | 오류·중복 사전 제거, 정합성 확보 | 입력 시점 차단으로 오류 유입 최소화 |
| **비용 절감** | 사후 정제·재작업 비용 감소 | 늦게 고칠수록 커지는 수정비용 회피 |
| **활용성 제고** | 개방·연계 데이터의 신뢰성·재사용성↑ | 표준화로 기관 간 연계·통합 용이 |

## 5. 고려사항 및 시사점
- **데이터 표준화·거버넌스 체계와의 연계**: 매뉴얼의 표준단어·도메인 사전은 전사 데이터 거버넌스(표준 관리 조직·프로세스) 위에서 유지·갱신되어야 실효성이 있다. 표준을 만들어도 관리 주체가 없으면 시간이 지나며 다시 흐트러진다.
- **개방·마이데이터·AI 학습데이터의 품질 기반**: 공공데이터가 민간 서비스와 AI 학습에 재사용되면서, 원천 품질이 곧 파생 서비스 품질을 좌우한다. "쓰레기가 들어가면 쓰레기가 나온다(GIGO)"는 문제를 상류에서 막는다는 점에서 전략적 가치가 크다.
- **자동화와 조직**: 진단 항목을 자동 진단 도구로 정기 실행하고, 품질관리 전담 조직·역할을 두어 예방-진단-개선의 순환을 제도화한다.
- **트레이드오프**: 초기 표준 수립·모델링에 시간과 인력이 더 들지만, 이는 사후 정제·오류 확산 비용에 비하면 훨씬 저렴한 선행 투자다.

---

> **한 줄 요약**: 공공DB 표준화 관리 매뉴얼은 *구축 단계(분석~운영)마다 예방적 품질관리 활동*을 배치하고 *표준화·모델·값·구조무결성의 4개 진단영역 9개 진단항목*으로 사전 진단하여, 오류가 발생·확산되기 전에 차단하고 개방·연계·AI 학습데이터의 품질 기반을 확보한다.

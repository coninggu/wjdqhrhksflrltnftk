# 리팩토링(Refactoring)과 디자인 패턴(Design Pattern)

## 1. 개요

### 가. 정의
> **리팩토링**은 소프트웨어의 **외부 동작(기능)은 그대로 유지하면서 내부 구조를 개선**하는 활동이고, **디자인 패턴**은 자주 발생하는 설계 문제에 대한 **검증된 재사용 가능한 해결책**이다.

두 기법의 관계를 이해하는 핵심은 '**리팩토링은 과정, 디자인 패턴은 목표(지향점)**'라는 점이다. 코드가 시간이 지나며 지저분해지고 이해·수정이 어려워질 때(코드 스멜), 리팩토링으로 내부를 정리한다. 이때 "어떤 구조로 개선할 것인가"의 방향을 제시하는 것이 디자인 패턴이다. 즉 리팩토링을 하다 보면 자연스럽게 검증된 패턴 구조로 수렴하고, 반대로 패턴을 적용하는 과정이 곧 리팩토링이 된다. 둘의 공통점은 소프트웨어 품질(유지보수성·유연성·재사용성)을 높인다는 것이고, 차이점은 리팩토링이 '**행위(개선 활동)**'라면 디자인 패턴은 '**산출물(해결 구조)**'이라는 데 있다.

### 나. 필요성
소프트웨어는 요구 변경으로 끊임없이 수정되며, 방치하면 구조가 부패해 유지보수 비용이 폭증한다. 리팩토링과 디자인 패턴은 코드를 지속적으로 건강하게 유지하는 두 축이다.

## 2. 리팩토링

리팩토링은 기능을 바꾸지 않으므로, 안전하게 하려면 **테스트가 전제**되어야 한다. 순서는 '작은 단위로 코드를 고치고 → 테스트로 기능이 그대로인지 확인'을 반복하는 것이다. 리팩토링을 촉발하는 신호가 **코드 스멜(Code Smell)** 로, 중복 코드·긴 메서드·큰 클래스·과도한 매개변수 등 "고쳐야 할 냄새가 나는" 코드를 말한다.

| 항목 | 내용 |
|---|---|
| **목적** | 기능 유지, 내부 구조 개선(가독성·유지보수성) |
| **순서** | 테스트 확보 → 소단위 개선 → 테스트 검증 반복 |
| **주요 기법** | 메서드 추출, 이름 변경, 클래스 분리, 조건문 단순화 |
| **코드 스멜** | 중복 코드, 긴 메서드, 큰 클래스, 과도한 매개변수 |

예를 들어 여러 곳에 중복된 코드(코드 스멜)는 **메서드 추출**로 하나로 모으고, 긴 메서드는 여러 작은 메서드로 나눈다.

## 3. 디자인 패턴

디자인 패턴은 GoF(Gang of Four)가 정리한 것이 대표적이며, 목적에 따라 생성·구조·행위 패턴으로 나뉜다.

| 유형 | 목적 | 예 |
|---|---|---|
| **생성 패턴** | 객체 생성 캡슐화 | 싱글턴, 팩토리, 빌더 |
| **구조 패턴** | 객체·클래스 조합 | 어댑터, 데코레이터, 프록시 |
| **행위 패턴** | 객체 간 책임·상호작용 | 옵서버, 전략, 커맨드 |

## 4. 공통점과 차이점

| 구분 | 리팩토링 | 디자인 패턴 |
|---|---|---|
| **성격** | 행위(개선 활동) | 산출물(해결 구조) |
| **대상** | 기존 코드 개선 | 설계 문제 해결책 |
| **공통점** | 품질(유지보수성·유연성·재사용성) 향상 |  |
| **관계** | 패턴을 지향점 삼아 개선 | 리팩토링으로 도달 |

## 5. 고려사항 및 시사점

1. **테스트 없는 리팩토링은 위험**하다. 기능이 그대로임을 보장하려면 자동화된 테스트가 전제되어야 하며, TDD와 결합하면 안전한 지속 개선이 가능하다.
2. **패턴 남용은 과설계**다. 문제가 없는데 패턴을 억지로 적용하면 오히려 복잡도만 늘리므로, 코드 스멜이 실제로 나타날 때 필요한 패턴을 절제해 도입한다.
3. **지속적 개선 문화**가 핵심이다. 리팩토링을 별도 작업이 아니라 개발의 일상적 습관으로 내재화하고, CI에 코드 품질 점검을 통합해 코드 부패를 예방한다.

---

> **한 줄 요약**: 리팩토링은 *기능 유지하며 내부 구조를 개선하는 활동*, 디자인 패턴은 *검증된 설계 해결책* 으로, 코드 스멜을 테스트 기반 리팩토링으로 개선하며 패턴을 지향점 삼아 품질을 높이되 패턴 남용은 경계한다.

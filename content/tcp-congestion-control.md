# TCP 혼잡 제어(Congestion Control)

## 1. 개요

### 가. 정의
> 네트워크에 **혼잡(패킷 과부하)이 발생했을 때 송신 속도를 조절**해 혼잡을 완화하는 TCP의 메커니즘. 흐름 제어(수신측 속도)와 달리 네트워크 전체의 부하를 다룬다.

혼잡 제어가 필요한 이유는 '**혼잡 붕괴(Congestion Collapse)** 방지'다. 여러 송신자가 통제 없이 데이터를 쏟아내면 라우터 큐가 넘쳐 패킷 손실→재전송→더 큰 혼잡의 악순환이 생긴다. TCP는 손실을 혼잡 신호로 해석해 스스로 전송량을 줄여 네트워크를 보호한다.

## 2. 혼잡 제어 메커니즘 구성요소 (1)

```mermaid
flowchart LR
  A[슬로우 스타트] --> B[혼잡 회피] --> C[빠른 재전송] --> D[빠른 회복]
  style A fill:#e8f0fe,stroke:#2f6fed,stroke-width:2px
```

| 구성요소 | 내용 |
|---|---|
| **슬로우 스타트** | cwnd를 지수적으로 증가(1→2→4…), 임계값(ssthresh)까지 |
| **혼잡 회피** | 임계값 이후 선형(1씩) 증가로 신중히 |
| **빠른 재전송** | 중복 ACK 3회 시 타임아웃 전 즉시 재전송 |
| **빠른 회복** | 재전송 후 슬로우스타트 대신 혼잡회피로 복귀 |

## 3. 혼잡 상황 감지 (2)

| 신호 | 의미 |
|---|---|
| **타임아웃(RTO)** | 심각한 혼잡(패킷 손실) → cwnd 초기화 |
| **중복 ACK 3회** | 경미한 손실 → 빠른 재전송·회복 |
| **ECN(명시적 표시)** | 라우터가 혼잡을 비트로 통지(손실 전 대응) |

## 4. 혼잡 상황 제어 (3) 및 혼잡 윈도우(cwnd) 크기 (4)

- **혼잡 윈도우(cwnd)**: 송신 가능한 데이터량을 나타내는 창으로, 혼잡 상태에 따라 동적 조절
- **실제 전송량** = min(cwnd, rwnd) — 혼잡 윈도우와 수신 윈도우 중 작은 값
- **제어**: 손실 감지 시 ssthresh를 cwnd의 절반으로, cwnd를 줄여(타임아웃 시 1로) 다시 증가

| 알고리즘 | 특징 |
|---|---|
| **Tahoe** | 손실 시 슬로우스타트로 |
| **Reno** | 빠른 회복 도입 |
| **CUBIC** | 고대역·고지연에 최적(리눅스 기본) |
| **BBR** | 대역폭·RTT 추정 기반(구글) |

## 5. 시사점
- 손실 기반(Reno·CUBIC)에서 **모델 기반(BBR)** 으로 진화
- 데이터센터·고속망 특성에 맞는 알고리즘 선택
- ECN 등 능동적 혼잡 신호로 손실 전 선제 대응

---

> **한 줄 요약**: TCP 혼잡 제어는 *슬로우스타트·혼잡회피·빠른재전송·빠른회복* 으로 혼잡 윈도우(cwnd)를 조절하며, 타임아웃·중복ACK로 혼잡을 감지해 전송량을 줄여 혼잡 붕괴를 방지한다.

# TCP 혼잡 제어(Congestion Control)

## 1. 개요

### 가. 정의
> 네트워크에 **혼잡(패킷 과부하)이 발생했을 때 송신 속도를 조절해 혼잡을 완화**하는 TCP의 메커니즘. 수신측의 처리 속도에 맞추는 흐름 제어(Flow Control)와 달리, 네트워크 전체의 부하를 다룬다.

혼잡 제어가 필요한 근본 이유는 '**혼잡 붕괴(Congestion Collapse)의 방지**'다. 여러 송신자가 네트워크 상태를 무시하고 데이터를 마구 쏟아내면, 중간 라우터의 큐가 넘쳐 패킷이 버려진다. 그러면 송신자들은 손실된 패킷을 재전송하고, 이 재전송이 다시 혼잡을 키우는 악순환이 벌어져 결국 네트워크 처리량이 0에 수렴하는 붕괴가 일어난다. TCP는 이를 막기 위해 영리한 가정을 한다. 무선 오류가 드문 유선망에서는 **패킷 손실을 곧 혼잡의 신호로 해석**하고, 손실이 감지되면 스스로 전송량을 줄인다. 즉 개별 종단(TCP)이 협조적으로 속도를 조절해 네트워크라는 공유 자원을 보호하는 것이다.

### 나. 필요성
인터넷은 수많은 종단이 공유하는 자원이므로, 각자가 자기 속도만 고집하면 전체가 붕괴한다. 혼잡 제어는 종단들이 자율적으로 협력해 네트워크를 안정적으로 유지하는 핵심 장치다.

## 2. 혼잡 제어 메커니즘 구성요소

```mermaid
  style A fill:#e8f0fe,stroke:#2f6fed,stroke-width:2px
```

혼잡 제어는 네 요소로 동작한다. **슬로우 스타트** 는 연결 초기에 혼잡 윈도우(cwnd)를 1에서 지수적으로(1→2→4→8) 빠르게 키워 가용 대역을 탐색하되, 임계값(ssthresh)에 도달하면 멈춘다. **혼잡 회피** 는 임계값 이후 윈도우를 1씩 선형 증가시켜 신중하게 대역을 늘린다. **빠른 재전송** 은 중복 ACK가 3번 오면 타임아웃을 기다리지 않고 즉시 손실 패킷을 재전송한다. **빠른 회복** 은 재전송 후 슬로우스타트로 되돌아가지 않고 혼잡 회피 상태로 복귀해 급격한 속도 저하를 막는다.

| 구성요소 | 내용 |
|---|---|
| **슬로우 스타트** | cwnd 지수 증가(1→2→4…), ssthresh까지 |
| **혼잡 회피** | 임계값 이후 선형(1씩) 증가 |
| **빠른 재전송** | 중복 ACK 3회 시 즉시 재전송 |
| **빠른 회복** | 재전송 후 혼잡회피로 복귀 |

## 3. 혼잡 상황 감지

TCP는 두 가지 신호로 혼잡을 감지한다. **타임아웃(RTO)** 은 응답이 오랫동안 없는 심각한 손실로, 강한 혼잡을 의미해 윈도우를 크게 줄인다(1로 초기화). **중복 ACK 3회** 는 일부 패킷만 손실된 경미한 상황으로, 빠른 재전송·회복으로 대응한다. 최근에는 손실이 나기 전에 라우터가 혼잡을 비트로 미리 알려주는 **ECN(명시적 혼잡 통지)** 도 활용된다.

| 신호 | 의미 |
|---|---|
| **타임아웃(RTO)** | 심각한 손실 → cwnd 초기화 |
| **중복 ACK 3회** | 경미한 손실 → 빠른 재전송·회복 |
| **ECN** | 라우터가 손실 전 혼잡 통지 |

## 4. 혼잡 상황 제어와 혼잡 윈도우(cwnd)

핵심 변수인 **혼잡 윈도우(cwnd)** 는 아직 확인받지 않은 채 보낼 수 있는 데이터량, 즉 전송 속도를 나타낸다. 실제 전송량은 혼잡 윈도우와 수신 윈도우(rwnd) 중 **작은 값**으로 결정된다(둘 다 만족해야 하므로). 혼잡이 감지되면 임계값(ssthresh)을 현재 cwnd의 절반으로 낮추고 cwnd를 줄인 뒤 다시 서서히 늘려간다. 이 '증가하다 손실 나면 감소'를 반복하는 톱니(sawtooth) 형태가 TCP 혼잡 제어의 특징이다.

| 알고리즘 | 특징 |
|---|---|
| **Tahoe** | 손실 시 슬로우스타트로 복귀 |
| **Reno** | 빠른 회복 도입 |
| **CUBIC** | 고대역·고지연에 최적(리눅스 기본) |
| **BBR** | 대역폭·RTT 추정 기반(구글) |

## 5. 고려사항 및 시사점

1. **손실 기반에서 모델 기반으로 진화**하고 있다. 전통적 Reno·CUBIC은 손실을 혼잡 신호로 삼지만, 구글의 BBR은 대역폭과 지연을 직접 추정해 손실 없이도 최적 속도를 찾는다.
2. **네트워크 특성에 맞는 알고리즘 선택**이 중요하다. 데이터센터의 저지연 고대역 환경과 무선·위성의 고손실 환경은 최적 알고리즘이 다르므로, 환경에 맞게 조정한다.
3. **능동적 혼잡 신호(ECN)로 손실 전 대응**하는 방향으로 발전한다. 패킷을 버려서 알리는 대신 미리 통지받아 선제 조절하면, 재전송을 줄이고 지연을 낮출 수 있다.

---

> **한 줄 요약**: TCP 혼잡 제어는 *슬로우스타트·혼잡회피·빠른재전송·빠른회복* 으로 혼잡 윈도우(cwnd)를 톱니 형태로 조절하며, 타임아웃·중복ACK로 혼잡을 감지해 전송량을 줄여 혼잡 붕괴를 방지하고 BBR·ECN으로 진화한다.

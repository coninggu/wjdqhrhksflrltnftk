# 스레싱(Thrashing)

## 1. 개요

### 가. 정의
> **스레싱**은 가상 메모리 시스템에서 **페이지 부재(Page Fault)가 과도하게 발생해, CPU가 실제 작업보다 페이지 교체(스와핑)에 더 많은 시간을 쓰면서 시스템 성능이 급격히 떨어지는 현상**이다.

스레싱의 본질은 '**일은 안 하고 페이지 바꾸기만 반복한다**'는 데 있다. 가상 메모리는 프로그램의 일부만 물리 메모리에 올려두고, 필요한 페이지가 없으면(페이지 부재) 디스크에서 가져온다. 그런데 물리 메모리에 비해 실행하려는 프로세스가 너무 많으면, 각 프로세스가 최소한의 페이지도 확보하지 못한다. 그러면 A 페이지를 올리려 B를 내리고, 곧 B가 필요해 다시 올리려 A를 내리는 일이 끝없이 반복된다. CPU는 정작 명령을 실행하지 못하고 디스크 입출력만 기다린다. 결과적으로 CPU 이용률은 바닥을 치는데, OS는 이를 '한가하다'고 오판해 프로세스를 더 투입하고, 스레싱은 더 악화된다. 이 악순환이 스레싱의 무서운 점이다. 근본 원인은 **다중 프로그래밍 정도(degree of multiprogramming)가 과도**한 것이며, 해법도 이를 적절히 통제하는 데 있다.

## 2. 원인과 해결 방법

스레싱을 막는 핵심 개념이 **워킹셋(Working Set)** 과 **페이지 부재 빈도(PFF)** 다. 워킹셋은 프로세스가 최근 일정 시간 동안 참조한 페이지 집합으로, 이 만큼의 프레임을 확보해줘야 스레싱을 피한다. PFF 기법은 페이지 부재율을 감시해, 너무 높으면 프레임을 더 주고 너무 낮으면 회수한다.

| 해결 방법 | 내용 |
|---|---|
| **워킹셋 모델** | 프로세스가 필요로 하는 페이지 집합만큼 프레임 할당 |
| **PFF(Page-Fault Frequency)** | 페이지 부재율 상·하한으로 프레임 동적 조절 |
| **다중프로그래밍 정도 조절** | 과다 시 일부 프로세스 스왑아웃(중단) |
| **물리 메모리 증설** | 근본적 프레임 부족 해소 |
| **지역성 활용** | 참조 지역성 높은 프로그램 구조 |

## 3. 관련 개념 — 지역성(Locality)

스레싱 관리의 이론적 근거는 **참조 지역성**이다. 프로그램은 특정 시간엔 일부 페이지만 집중 참조하는(시간·공간 지역성) 경향이 있다. 이 지역성 집합(워킹셋)만큼만 메모리에 올려두면 페이지 부재가 적어진다. 즉 지역성을 이해하고 워킹셋을 보장하는 것이 스레싱 예방의 핵심이다.

## 4. 고려사항 및 시사점

1. **CPU 이용률만 보고 판단하면 위험**하다. 스레싱 시 CPU 이용률이 낮게 보여 OS가 프로세스를 더 투입하면 악순환에 빠지므로, 페이지 부재율·스왑 활동을 함께 모니터링해야 한다.
2. **워킹셋 보장이 근본 대책**이다. 프로세스가 필요로 하는 최소 프레임(워킹셋)을 확보해주는 것이 스레싱 예방의 핵심이며, 부족하면 다중프로그래밍 정도를 낮춘다.
3. **현대 환경에서도 유효**하다. 컨테이너·가상화 환경에서 메모리 오버커밋(over-commit) 시 유사한 스와핑 폭주가 발생하므로, 자원 한계(limit) 설정과 메모리 모니터링이 중요하다.

---

> **한 줄 요약**: 스레싱은 *과도한 다중프로그래밍으로 페이지 부재가 폭증해 CPU가 스와핑만 반복하며 성능이 급락하는 현상* 으로, 워킹셋 보장·PFF·다중프로그래밍 정도 조절로 해결하며 CPU 이용률 오판에 주의해야 한다.
